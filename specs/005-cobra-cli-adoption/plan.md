# Implementation Plan: Modern CLI Interface

**Branch**: `005-cobra-cli-adoption` | **Date**: 2026-06-10 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/005-cobra-cli-adoption/spec.md`

## Summary

Rune already depends on Cobra (`github.com/spf13/cobra v1.10.2`) but uses it as a
single root command that hand-dispatches `mcp`/`serve`/`completion` by inspecting
positional arguments, hand-parses the server's flags in a manual loop, and wraps
completion in a custom helper. This feature adopts Cobra **idiomatically** to deliver
a modern, self-describing CLI:

- `serve` (with `mcp` as a documented alias), `version`, and Cobra's auto-generated
  `completion`/`help` become real, discoverable subcommands, each with its own help,
  flags, and examples.
- The server's options become first-class, validated flags.
- Shell completion is generated by Cobra's built-in machinery for bash/zsh/fish/
  powershell, and the **root's `ValidArgsFunction` dynamically completes Runefile
  task names with their doc summaries**.
- Mistyped commands get a "did you mean" suggestion.

The hard constraint (Constitution governance / backward-compatibility promise): the
externally observable contract — exit codes, stdout/stderr discipline, task argument
pass-through, spanned diagnostics, color rules, signal handling — is preserved
**exactly**. Tasks remain dynamic positional arguments; built-in command names take
precedence, and `rune -- <task>` is the escape hatch that keeps colliding task names
reachable. The existing integration suite is the regression guardrail.

## Technical Context

**Language/Version**: Go 1.25 (`go.mod`: `go 1.25.0`; Constitution floor: 1.24+).

**Primary Dependencies**: `github.com/spf13/cobra v1.10.2` (already present) and its
`github.com/spf13/pflag` (already indirect). `fatih/color`, `mattn/go-isatty` (already
used for color/TTY). **No new dependencies** — single-static-binary distribution
unchanged (Constitution V). Viper is explicitly **not** introduced (spec Assumptions).

**Storage**: N/A — configuration is the on-disk Runefile resolved by `config.Resolve`.

**Testing**: `go test ./...` and `-race` **inside Docker** (`docker-compose run --rm
test …`, never on host). Layered per Constitution VI: table-driven unit tests
(completion candidates, suggestion distance, serve flag validation) and end-to-end
integration tests that run the compiled binary and assert stdout, stderr, AND exit
code (help discoverability, completion output via `__complete`, serve flags, `--`
escape, did-you-mean).

**Target Platform**: Linux, macOS, Windows — single static binary; completion scripts
for bash, zsh, fish, powershell.

**Project Type**: Single-project CLI (command runner) with a public `mcpserver`
package; CLI wiring lives in `rune/` (package `main`), domain logic in `internal/cli`.

**Performance Goals**: `ValidArgsFunction` runs on every TAB — it MUST resolve+parse
the Runefile and return candidates quickly (target < 50 ms for a typical Runefile) and
MUST NOT run the full analyzer or block. Non-completion CLI startup overhead is
negligible.

**Constraints**: Backward-compatible CLI contract is sacrosanct (Governance).
`internal/cli` MUST stay framework-free — no `cobra` import there (Constitution VIII:
keep domain logic free of framework dependencies). Completion MUST never emit
errors/panics into the shell session (Constitution II).

**Scale/Scope**: Small fixed command set (`serve`+alias, `version`, auto `completion`/
`help`); Runefiles may define up to a few hundred tasks (completion candidate list).

### Source Code (repository root)

> **Path reconciliation (discovered during /speckit-tasks)**: the binary's `main`
> package currently lives in `rune/`, but the Runefile, `.goreleaser.yaml`, `Dockerfile`,
> and both test harnesses (`test/integration`, `test/docs`) all build `./cmd/rune` — a
> path that does not exist, so those suites cannot build today. This feature relocates
> the binary to `cmd/rune/` (matching all build tooling and the `golang-cli` `cmd/<app>/`
> layout). The tree below uses the post-relocation paths.

```text
cmd/rune/                     # package main — Cobra wiring (imports cobra + internal/cli)
├── main.go                   # MINIMAL: signal ctx, build newRootCmd, ExecuteContext, map exit code
├── root.go                   # NEW: newRootCmd() builder — global flags, RunE (task path),
│                             #      ValidArgsFunction (task-name completion), SilenceUsage/Errors,
│                             #      SetInterspersed(false), CompletionOptions
├── serve.go                  # NEW: newServeCmd() — `serve` (Aliases: ["mcp"]) with --http/--addr/
│                             #      --token-file/--mcp; RunE → cli.ServeMCP; flag-completion funcs
├── version.go                # NEW: newVersionCmd() — prints version+commit (parity with --version)
└── completion.go             # REMOVE custom genCompletion (replaced by Cobra's auto completion cmd)

internal/cli/                 # package cli — stays cobra-free (framework-independent)
├── complete.go               # NEW: TaskCandidates(opts) []TaskCandidate {Name, Doc} — resolve+parse
│                             #      Runefile, filter IsPrivate/osMatches; graceful (nil on any error)
├── suggest.go                # NEW: nearest(token, candidates) — small Levenshtein for did-you-mean
├── args.go                   # EDIT: splitArgs "unknown task" error → append did-you-mean suggestion
├── dispatch.go               # EDIT: Options gains Commands []string (reserved names for suggestions)
├── exit.go                   # UNCHANGED: exit-code contract is authoritative
└── serve.go / run.go / …     # UNCHANGED behavior

# Tests live beside code (rune/*_test.go, internal/cli/*_test.go) plus the
# existing end-to-end suite (compiled-binary integration tests).
```

**Structure Decision**: Single-project CLI. The binary is relocated `rune/` → `cmd/rune/`
(see path-reconciliation note above), and Cobra wiring is split out of the monolithic
`main.go` into per-command builder files under `cmd/rune/` (the `golang-cli` skill's
"one file per command" layout). Domain/completion data logic lives in `internal/cli`
and is imported by `main`; `internal/cli` does **not** import cobra, so the framework
boundary stays clean (Constitution VIII). `main` adapts `[]cli.TaskCandidate` →
`[]cobra.Completion` at the wiring layer.
